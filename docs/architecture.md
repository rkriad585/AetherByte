# AetherByte System Architecture

AetherByte is designed as a client-server web application, leveraging Flask for the backend and a combination of HTML, CSS (Tailwind CSS), and JavaScript (jQuery, CodeMirror) for the frontend. Its core innovation lies in its dynamic compiler and custom virtual machine, which together process a human-readable language, NaturalScript, and allow for extensible syntax definitions.

## High-Level Overview

The system follows a classic MVC-like pattern (Model-View-Controller), where:
*   **Client (Browser):** Renders the user interface, handles user input, and communicates with the server via AJAX requests.
*   **Server (Flask Application):** Manages routes, compiles NaturalScript, executes binary code, and interacts with the database.
*   **Database (SQLite):** Stores user scripts and shared project data.

## Component Breakdown

### 1. `app.py` - Application Entry Point

*   Initializes the Flask application.
*   Loads configuration from `config.py`.
*   Registers Flask Blueprints (e.g., `core_bp` from `core/routes.py`).
*   Initializes the SQLite database schema via `DatabaseManager`.
*   Starts the Flask development server.

### 2. `config.py` - Configuration Management

*   Defines application-wide settings such as `SECRET_KEY`, `DEBUG` mode, and `DB_NAME`.
*   Crucially, it defines `LANG_DIR`, specifying the directory (`Aetherlang`) where custom language definition JSON files are stored. This path is used by the `NaturalCompiler` to dynamically load syntax rules.

### 3. `core/` - Core Logic Module

This directory encapsulates the primary business logic of AetherByte.

#### `core/compiler.py` - Dynamic NaturalScript Compiler

*   **Purpose:** Translates NaturalScript code into a custom 8-bit binary instruction set.
*   **Dynamic Loading:** Scans the `Aetherlang` directory for `.json` files on initialization. Each JSON file defines a set of syntax rules.
*   **Rule Structure:** Each rule in a JSON file consists of:
    *   `name`: A descriptive name.
    *   `regex`: A regular expression to match NaturalScript commands, often using named capture groups for arguments.
    *   `opcode`: The 8-bit binary instruction code for the VM.
    *   `args`: A list defining how to extract and convert arguments from the regex match (e.g., `string` or `int`).
*   **Conversion Helpers:** Provides utilities to convert strings to binary (ASCII) and integers to 8-bit binary.
*   **Process:** Iterates through each line of NaturalScript, attempts to match it against loaded rules, extracts arguments, and constructs the corresponding binary instruction.

#### `core/vm.py` - 8-bit Virtual Machine

*   **Purpose:** Executes the 8-bit binary instruction set generated by the `NaturalCompiler`.
*   **Architecture:** A simple, register-less VM with a `memory` dictionary for variable storage and a `flag_cmp` for conditional logic (though conditionals are not fully implemented in the provided snippets, the flag suggests future expansion).
*   **Instruction Set:** Processes various opcodes for:
    *   Variable storage (`STORE`).
    *   Arithmetic operations (`ADD`, `SUB`, `MUL`, `DIV`).
    *   Output (`PRINT_STRING`, `PRINT_VAR`).
    *   Control flow (`LOOP`).
    *   System commands (`CLEAR`, `ALERT`).
*   **Execution Flow:** Reads binary data line by line, decodes opcodes and arguments, and performs the corresponding operations, accumulating output.

#### `core/routes.py` - Flask Blueprints and API Endpoints

*   **Purpose:** Defines all URL routes and their corresponding handler functions for the Flask application.
*   **Blueprints:** Uses a Flask Blueprint (`core_bp`) to organize routes.
*   **Frontend Routes:**
    *   `/`: Renders `index.html` (main IDE).
    *   `/settings`: Renders `settings.html`.
    *   `/shared/<share_id>`: Renders `index.html` with pre-loaded shared code and permissions.
*   **API Endpoints:**
    *   `/convert` (POST): Calls `NaturalCompiler.compile()`.
    *   `/run` (POST): Calls `BinaryVM.run()`.
    *   `/api/save`, `/api/list`, `/api/load/<id>`, `/api/delete/<id>`: Interact with `DatabaseManager` for script persistence.
    *   `/share` (POST): Creates a unique share ID and saves the code with a special title prefix for sharing.

#### `core/database.py` - SQLite Database Manager

*   **Purpose:** Provides an interface for interacting with the SQLite database (`aether_storage.db`).
*   **Connection Management:** Handles opening and closing database connections.
*   **Schema Initialization:** `init_db()` creates the `scripts` table if it doesn't exist.
*   **CRUD Operations:** Implements methods for:
    *   `save_script(title, code)`: Inserts new scripts.
    *   `get_all_scripts()`: Retrieves metadata for all saved scripts.
    *   `get_script_by_id(script_id)`: Fetches the code content of a specific script.
    *   `delete_script(script_id)`: Removes a script.

### 4. `templates/` - HTML User Interface

*   **`base.html`:** The base template, defining the overall page structure, including common headers, footers, meta tags, and linking external resources (Tailwind CSS, jQuery, CodeMirror). It uses Jinja2 blocks for content injection.
*   **`index.html`:** The main interactive IDE. It contains the CodeMirror editor, binary output panel, terminal output, and modals for saving, loading, and sharing code. It integrates JavaScript for dynamic behavior.
*   **`settings.html`:** Provides controls for user preferences, allowing customization of the editor and application layout.

### 5. `static/` - Frontend Assets

*   **`script.js`:** The primary JavaScript file for client-side logic.
    *   Initializes CodeMirror editor.
    *   Handles AJAX calls to backend API endpoints (`/convert`, `/run`, `/api/save`, etc.).
    *   Manages UI interactions (button clicks, modal display).
    *   Applies and persists user settings using `localStorage`.
    *   Includes logic for binary animation, terminal output rendering, and sharing functionality.
*   **`images/`:** Contains static image assets like the project logo.

## Data Flow Example: Compiling and Running Code

1.  **User Input:** A user types NaturalScript code into the CodeMirror editor in `index.html`.
2.  **Compile Request:** The user clicks the `COMPILE` button. `static/script.js` captures the editor content and sends it via an AJAX POST request to `/convert`.
3.  **Server-Side Compilation:**
    *   `core/routes.py` receives the request and calls `compiler.compile()` from `core/compiler.py`.
    *   `compiler.py` loads its rules from `Aetherlang/*.json`, parses the NaturalScript, and generates the 8-bit binary code.
    *   The binary code is returned as a JSON response to the client.
4.  **Display Binary:** `static/script.js` receives the binary code and displays it in the "BINARY CODE" panel.
5.  **Run Request:** The user clicks the `RUN` button. `static/script.js` sends the *binary code* (not the NaturalScript) via an AJAX POST request to `/run`.
6.  **Server-Side Execution:**
    *   `core/routes.py` receives the request and calls `vm.run()` from `core/vm.py`.
    *   `vm.py` executes the binary instructions, updates its internal memory, and generates a list of output strings.
    *   The output is returned as a JSON response to the client.
7.  **Display Output:** `static/script.js` receives the output and appends it to the "TERMINAL OUTPUT" panel, applying styling for errors or system messages.

## Key Technologies

*   **Backend:** Python, Flask, SQLite
*   **Frontend:** HTML5, CSS3 (Tailwind CSS), JavaScript (jQuery), CodeMirror (for the editor)
*   **Language Definition:** JSON

This architecture provides a modular and extensible foundation for AetherByte, allowing for easy expansion of both the NaturalScript language and the application's features.
