# System Design: AetherByte Architecture 🏗️

This document provides a comprehensive overview of the AetherByte system architecture, detailing its modular components, data flow, and underlying technologies.

## 1. Project Structure 📁

The AetherByte project is organized into a clear, modular structure to enhance maintainability, scalability, and separation of concerns.

```
.
├── Aetherlang/                 # 📂 Language definition files (JSON) for the dynamic compiler.
│   └── standard.json           #    Default NaturalScript syntax rules.
├── config.py                   # ⚙️ Application-wide configuration settings.
├── app.py                      # 🚀 Main Flask application entry point.
├── core/                       # 🧠 Core backend logic and services.
│   ├── compiler.py             #    NaturalScript to 8-bit binary compiler.
│   ├── database.py             #    SQLite database management for projects.
│   ├── routes.py               #    Flask Blueprints defining API and web routes.
│   └── vm.py                   #    Custom 8-bit Virtual Machine for binary execution.
├── docs/                       # 📚 Project documentation.
│   ├── architecture/           #    Architectural documentation.
│   │   └── system-design.md    #       This document.
│   ├── api/                    #    API documentation.
│   │   └── reference.md        #       API endpoint reference.
│   └── guides/                 #    Developer guides.
│       └── contribution.md     #       Contribution guidelines.
├── static/                     # 🖼️ Static assets (CSS, JS, images).
│   ├── images/                 #    Application images/icons.
│   └── script.js               #    Frontend JavaScript for IDE functionality.
├── templates/                  # 🌐 Jinja2 HTML templates for the web interface.
│   ├── base.html               #    Base template for common UI elements.
│   ├── index.html              #    Main IDE page.
│   └── settings.html           #    User settings page.
└── README.md                   # 📄 Project overview and quick start guide.
```

## 2. Core Components Overview 🧩

### 2.1. `config.py`
This file centralizes application configuration.
| Property | Type | Description |
| :------- | :--- | :---------- |
| `SECRET_KEY` | `str` | Flask secret key for session management and security. |
| `DEBUG` | `bool` | Enables/disables debug mode for Flask. |
| `DB_NAME` | `str` | Name of the SQLite database file (`aether_storage.db`). |
| `LANG_DIR` | `str` | Path to the directory containing NaturalScript language definition JSON files. |

### 2.2. `app.py`
The application factory for the Flask app. It initializes the Flask instance, loads configuration from `config.py`, registers blueprints (`core_bp`), and ensures the database schema is initialized via `DatabaseManager`.

### 2.3. `core/compiler.py`
The dynamic compiler for NaturalScript.
- **Dynamic Rule Loading:** Scans the `Aetherlang/` directory for JSON files, each defining syntax rules (regex, opcode, arguments).
- **Compilation Logic:** Iterates through input NaturalScript lines, matches them against loaded rules using regular expressions, extracts arguments, and converts them into an 8-bit binary instruction set.
- **Binary Conversion:** Provides utility functions to convert strings and integers into their 8-bit binary representations.

### 2.4. `core/vm.py`
The custom 8-bit Virtual Machine.
- **Instruction Set:** Executes a predefined set of 8-bit binary opcodes generated by the compiler.
- **Memory Management:** Manages a simple key-value store (`self.memory`) for variables and `self.lists` for list data structures.
- **Control Flow:** Supports basic control flow like loops and conditional execution based on comparison flags.
- **Output Handling:** Collects execution output, including print statements, system alerts, and error messages.

### 2.5. `core/database.py`
Manages interactions with the SQLite database (`aether_storage.db`).
- **Schema Initialization:** Creates the `scripts` table if it doesn't exist.
- **CRUD Operations:** Provides methods for saving, retrieving, listing, and deleting NaturalScript projects.
- **Row Factory:** Configures connections to return rows as dictionaries for easier access.

### 2.6. `core/routes.py`
Defines the Flask routes and API endpoints.
- **Web Routes:** Renders `index.html` (IDE) and `settings.html`.
- **Compiler API:** `/convert` (POST) for compiling NaturalScript.
- **VM API:** `/run` (POST) for executing compiled binary code.
- **Database API:** `/api/save`, `/api/list`, `/api/load/<id>`, `/api/delete/<id>` for project management.
- **Sharing Feature:** `/share` (POST) to generate unique shareable links and `/shared/<id>` (GET) to load shared projects with specific permissions.

### 2.7. Frontend (`templates/`, `static/script.js`)
- **`templates/index.html`:** The main IDE interface, integrating CodeMirror for code editing, displaying compiled binary, and terminal output.
- **`templates/settings.html`:** Provides user-configurable options for the IDE (font size, layout, etc.) stored in `localStorage`.
- **`static/script.js`:** Handles all client-side logic, including:
    - CodeMirror initialization and interaction.
    - AJAX calls to backend API endpoints (compile, run, save, load, share).
    - UI updates based on user settings and backend responses.
    - LocalStorage management for user preferences.

### Data Flow Explanation:

1.  **User Input:** A user writes NaturalScript code in the CodeMirror editor on `index.html`.
2.  **Compilation Request:** When the "COMPILE" button is clicked, `script.js` sends the NaturalScript code via an AJAX POST request to the `/convert` endpoint in `core/routes.py`.
3.  **Compilation Process:** `core/routes.py` invokes `core/compiler.py`. The compiler loads its language definition rules from JSON files in `Aetherlang/`, parses the NaturalScript, and generates 8-bit binary code.
4.  **Binary Display:** The compiled binary code is returned to `script.js`, which then displays it in the binary output panel.
5.  **Execution Request:** When the "RUN" button is clicked, `script.js` sends the *compiled binary code* via an AJAX POST request to the `/run` endpoint in `core/routes.py`.
6.  **VM Execution:** `core/routes.py` invokes `core/vm.py`. The VM executes the binary instructions, managing its internal memory and producing output (e.g., print statements, errors).
7.  **Output Display:** The VM's execution output is returned to `script.js`, which displays it in the terminal panel.
8.  **Project Management:**
    *   "SAVE" functionality sends code and title to `/api/save`.
    *   "LOAD" functionality fetches a list of projects from `/api/list` and then a specific project's code from `/api/load/<id>`.
    *   "DELETE" functionality sends a request to `/api/delete/<id>`.
    *   All these interactions are handled by `core/routes.py` which delegates to `core/database.py` for SQLite operations on `aether_storage.db`.
9.  **Sharing:** The "SHARE" feature sends the current code and desired permission to `/share`, which generates a unique ID, saves the project (with a special title prefix) to the database, and returns a shareable URL. Accessing this URL (`/shared/<id>`) loads the project directly into the IDE.
10. **User Settings:** User preferences (font size, layout, etc.) are managed client-side by `script.js` and stored in the browser's `localStorage` for persistence across sessions. The `settings.html` page provides the UI for these configurations.

## 4. Caveats and Edge Cases ⚠️

*   **Security:** The current VM and compiler are designed for a controlled environment. Direct execution of arbitrary user-defined opcodes or complex system interactions is not supported. Input validation is primarily at the regex level in the compiler; robust security checks for VM inputs are limited.
*   **Scalability:** SQLite is suitable for single-instance, local project storage. For multi-user or high-volume scenarios, a more robust database (e.g., PostgreSQL, MySQL) and a distributed architecture would be required.
*   **Error Handling:** While basic error reporting exists in the VM and compiler, comprehensive error recovery and detailed debugging information are areas for future enhancement.
*   **Language Extensibility:** The dynamic compiler relies on well-formed JSON rules. Malformed or conflicting rules can lead to unexpected compilation behavior or errors.
*   **VM Limitations:** The 8-bit VM has inherent limitations in data types (e.g., integer range), memory capacity, and instruction complexity. It is not a general-purpose VM.
*   **Share Feature:** The current sharing mechanism reuses the `scripts` table with a special title prefix. For a production system, a dedicated `shares` table with proper access control and expiry mechanisms would be more appropriate.
